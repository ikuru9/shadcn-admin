# 계획 검토 및 정리

`writing-plans`로 작성한 계획을 구현 전에 검토하고, 빠진 내용, 모호한 단계, 검증 누락, 범위 초과를 정리한다.

## 목적

- 계획이 바로 구현 가능한 상태인지 확인한다.
- 작업 순서가 TDD에 맞는지 확인한다.
- 테스트, 검증, 경계 조건, 파일 경로가 빠지지 않았는지 확인한다.
- 구현 전에 계획을 더 작은 작업 단위로 정리한다.

## 검토 항목

1. 요구사항이 모두 계획에 반영됐는지 확인한다.
2. 각 작업이 2~5분 단위인지 확인한다.
3. 각 작업에 정확한 파일 경로가 있는지 확인한다.
4. 테스트 파일 또는 테스트 전략이 포함됐는지 확인한다.
5. 검증 명령이 포함됐는지 확인한다.
6. 생성 코드 직접 수정, 경로 변경, API 계약 변경 같은 위험이 드러나는지 확인한다.
7. TDD 순서가 `RED -> GREEN -> REFACTOR`로 표현되는지 확인한다.

## 정리 규칙

- 모호한 표현을 제거한다.
- 빠진 테스트 단계를 보완한다.
- 구현 전에 필요한 선행 작업을 앞쪽으로 옮긴다.
- 범위를 벗어난 단계는 제거하거나 별도 후속 작업으로 분리한다.
- 자리표시자, TBD, TODO를 남기지 않는다.

## 결과물

검토 결과는 다음 중 하나여야 한다.

- 구현 가능: 현재 계획으로 바로 `test-driven-development`에 들어간다.
- 수정 필요: 계획을 고친 뒤 다시 검토한다.

## 인계

계획 검토가 끝나면 `test-driven-development`로 넘어간다.
