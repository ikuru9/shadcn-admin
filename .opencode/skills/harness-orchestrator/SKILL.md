# Harness Orchestrator

## 목적

작업을 `frontend`, `integrator`, `qa`로 나눠서 충돌을 줄이고, 검증 가능한 순서로 실행한다.
이 저장소에서는 route tree, provider, generated client, MSW mock, test script가 연결되도록 조율한다.

## 실행 모드

에이전트 팀 모드를 기본으로 사용한다. UI와 데이터 연결, 검증이 함께 움직여야 하므로 중간 결과를 서로 확인하게 한다.

## Phase 1: 컨텍스트 확인

- 기존 변경 파일이 있으면 읽는다.
- 사용자가 새 기능을 요청했는지, 기존 작업을 수정하라고 했는지, 일부만 다시 하라고 했는지 구분한다.
- 영향 범위를 좁혀 불필요한 재작업을 피한다.

## Phase 2: 분해

- 화면 변화는 `frontend`로 보낸다.
- 데이터/상태/검증은 `integrator`로 보낸다.
- 마지막에 `qa`가 연결부를 확인한다.
- generated code가 필요하면 `integrator`가 먼저 보고, UI는 그 뒤에 맞춘다.

## Phase 3: 실행 순서

1. 관련 파일과 현재 구현을 확인한다.
2. 가장 작은 안전한 변경부터 적용한다.
3. 각 변경 뒤에 연결된 코드 경로를 확인한다.
4. QA 결과가 나오면 필요한 부분만 다시 할당한다.
5. API 계약이 바뀌면 mock과 생성 코드를 동기화한다.

## 에러 핸들링

- 정보가 부족하면 한 번만 질문한다.
- 충돌이 생기면 원인 파일과 결과 파일을 함께 보고한다.
- 실패한 검증은 무시하지 말고 다음 작업의 입력으로 남긴다.

## 테스트 시나리오

- 정상 흐름: 새 UI 요구사항이 들어오면 화면 수정, 데이터 연결, 검증 순으로 진행한다.
- 에러 흐름: 기존 화면의 일부만 수정 요청이 오면 해당 파일과 연결된 훅만 다시 점검한다.
- 대규모 변경은 `pnpm typecheck`로 시작해 필요 시 `pnpm test`, `pnpm build`로 마무리한다.
